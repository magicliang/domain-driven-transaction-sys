## 项目结构
项目采用 Gradle 多模块架构，遵循 SOFA 分层思想，模块职责清晰、边界明确：
- biz-service-impl：业务服务实现层，WebFlux/Web MVC 支持，应用可启动模块
- biz-shared：业务共享模块，提供业务级别的共享组件
- common-dal：数据访问层，集成 MyBatis/JPA，支持多数据源
- common-service-facade：服务门面层，定义对外暴露的服务接口
- common-service-integration：服务集成层，处理第三方系统集成
- common-util：通用工具类库
- core-model：核心领域模型，包含实体、值对象、聚合根等
- core-service：核心服务层，实现业务逻辑

## 性能考量
- 并行测试与构建
  - Gradle 属性启用并行构建与缓存，提升整体构建效率。
  - 测试任务根据 CPU 核数设置最大并行进程数，加速测试执行。
- 日志与可观测性
  - 通过 Log4j2 配置区分线上/线下日志级别，避免过度日志影响性能。
  - 引入 OpenTelemetry 依赖，便于统一埋点与追踪。
- 数据库连接池
  - HikariCP 连接池参数经过合理配置，平衡吞吐与延迟。
- 序列化性能
  - JSON 工具类提供缓存与禁用缓存两种策略，按场景选择以获得最佳性能。

## 故障排查指南
- 构建与测试
  - 若出现 Gradle 相关问题，参考项目提供的常见问题与修复指引。
  - 测试执行支持通配符与指定类/方法，便于定位问题。
- 数据库 Profile
  - 默认使用 Testcontainers 自动拉起 MariaDB 容器；若使用 Podman，需配置 socket 与镜像加速。
  - 外部数据库需在 application.yml 中正确配置连接参数。
- 部署与 K8s
  - 通过 K8s 清单与脚本一键部署/销毁环境；环境变量覆盖机制确保配置一致性。
  - 日志级别与 JVM 参数可通过 ConfigMap 调整，避免 OOM。

### E. GitHub SSH 免登录配置

**新增** 重要开发者体验改进

当您在推送代码时遇到 "Authentication failed" 或被要求输入用户名密码，这通常意味着您的远程仓库URL使用的是HTTPS格式而非SSH格式。

#### 配置步骤

1. **检查SSH密钥**
   ```bash
   ls -la ~/.ssh/
   # 应该能看到 id_rsa 和 id_rsa.pub
   ```

2. **复制公钥内容**
   ```bash
   cat ~/.ssh/id_rsa.pub
   ```

3. **添加公钥到GitHub**
   - 访问: https://github.com/settings/keys
   - 点击 "New SSH key"
   - 粘贴公钥内容并保存

4. **测试SSH连接**
   ```bash
   ssh -T git@github.com
   # 成功会显示: Hi username! You've successfully authenticated
   ```

5. **更新remote URL为SSH格式**
   ```bash
   # 查看当前remote URL
   git remote -v
   
   # 如果是HTTPS URL,改为SSH URL
   git remote set-url origin git@github.com:magicliang/domain-driven-transaction-sys.git
   
   # 验证修改
   git remote -v
   ```

#### URL格式对比

- ❌ **HTTPS**: `https://github.com/magicliang/domain-driven-transaction-sys.git` (需要输入密码)
- ✅ **SSH**: `git@github.com:magicliang/domain-driven-transaction-sys.git` (免密登录)

#### 常见问题解决

如果SSH配置后仍然出现问题：

1. **检查SSH代理**
   ```bash
   eval $(ssh-agent -s)
   ssh-add ~/.ssh/id_rsa
   ```

2. **验证Git配置**
   ```bash
   git config --global user.name "Your Name"
   git config --global user.email "your.email@example.com"
   ```

3. **检查防火墙设置**
   - 确保端口22未被防火墙阻止
   - 如果使用公司网络，可能需要配置代理

### F. 数据库查询API开发规范

**新增** 数据库查询API开发规范章节

#### 1. 表名白名单校验机制
- **正则校验**：表名必须符合 `^[a-zA-Z_][a-zA-Z0-9_]*$` 模式，确保只包含字母、数字、下划线且以字母或下划线开头
- **白名单缓存**：首次查询时从 `information_schema.tables` 加载所有用户表名，使用 `ConcurrentHashMap` 缓存
- **懒加载策略**：避免启动时的数据库查询开销，首次使用时才加载白名单

#### 2. SQL注入防护机制
- **动态SQL构建**：使用 MyBatis `@SelectProvider` 注解，通过 `DatabaseCountSqlProvider` 动态构建 SQL
- **参数化限制**：由于 MyBatis 不支持表名参数化，采用双重校验机制
- **安全验证**：正则校验 + 白名单校验，双重保险确保安全性

#### 3. 查询超时控制
- **超时阈值**：默认 5 秒超时，超过阈值记录警告日志
- **性能监控**：记录查询耗时，便于性能分析和优化
- **异常处理**：查询异常统一捕获并转换为运行时异常

#### 4. 错误处理与日志记录
- **参数验证**：使用 `@Valid` 注解和 `@NotBlank` 确保请求参数有效
- **统一异常处理**：通过 `ApiResult` 统一响应格式，包含成功状态、数据和错误信息
- **详细日志**：记录查询参数、结果和异常信息，便于问题排查

### G. 安全验证机制

**新增** 安全验证机制章节

#### 1. JWT 认证机制
- **令牌生成**：使用 HMAC256 算法签名，包含签发者、主题、声明、签发时间、过期时间和 JWT ID
- **令牌验证**：验证签发者、签名和过期时间，确保令牌有效性
- **声明设计**：JWT 中仅包含认证和授权必需信息，避免敏感数据泄露
- **过期策略**：短时有效的令牌配合服务端会话存储，平衡安全性和性能

#### 2. 内部 API 认证
- **拦截器实现**：`InnerApiAuthenticator` 实现 `AsyncHandlerInterceptor` 接口
- **认证逻辑**：提供统一的认证入口点，可根据业务需求扩展认证规则
- **顺序控制**：使用 `@Order` 注解控制拦截器执行顺序

#### 3. 统一异常处理
- **过滤器机制**：`StatusBasedFilter` 基于错误码的过滤器，处理客户端错误
- **响应格式**：统一的 API 响应格式，包含成功状态、数据和错误信息
- **状态管理**：支持健康检查等特殊路径的例外处理

#### 4. 安全最佳实践
- **密钥管理**：在真实项目中，密钥应从配置文件中读取，且更为复杂
- **令牌存储**：敏感信息不应放在 JWT 中，因为 JWT 可以被解码
